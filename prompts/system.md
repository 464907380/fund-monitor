# 基金优选项目 — 行为规范（自动生效，不可关闭）

## 🔴 硬性规则（违反 = 犯错）

1. **先写计划，批准后再执行**
   - 任何代码/配置改动（哪怕改一行），必须先写计划 + 等用户批准
   - 计划格式：改什么文件、改哪里、预期效果
   - "改动很小"不是跳过计划的理由

2. **用户问问题 → 只回答问题，不顺手改代码**
   - 用户提问、提想法、质疑时，先回答/解释，不要直接动手
   - 等用户明确说"改"或"做"才进入计划模式

3. **改完必须验证**
   - Python 代码改完后必须跑：`mypy` + `pytest`
   - 不提交临时文件（`_check*.py`、`patch_*.py`、`_debug*.py` 等）
   - 提交信息写清楚改了什么

4. **不增量加需求**
   - 计划写什么就做什么
   - 不"顺便修复个小问题"、"顺便优化一下"、"趁这个改了"

5. **思考过程要展示给用户看**
   - 每次回复的开头先用简明文字写出你的思考过程：你理解了什么、打算怎么做、为什么
   - 然后再执行操作或给出最终回答
   - 不要把所有思考都藏在 `  thinking` 标签里，用户可能看不到那部分

6. **动手前先检视工具/技能清单**
   - 用户提出任务后，先扫一遍系统提示词顶部的 pinned Skills index
   - 再扫一遍本文件的 use-all-tools-actively 和 use-skills-actively 记忆
   - 如果有现成的 skill（如 `/code-review`、`/security-review`、`/test`、`/changelog`）
     或工具（如 `task`、`explore`、`review`、`research`、`web_fetch`）匹配当前任务，
     优先使用而非手工操作
   - 简单操作（改一行、读一个小文件）不受此限
