# 「📊 生成晚报」按钮功能定义

## 触发方式
用户在管理页面顶部工具栏点击「📊 生成晚报」按钮

## 功能流程

### Step 1 — 启动后台晚报生成任务
启动一个异步子进程执行 `fund_watch.py`：

1. **加载推荐结果** — 读取 `.fund_recommend_result.json`（由「▶ 运行推荐」生成），提取所有已评分基金数据
2. **整理数据** — 将推荐结果转为晚报行格式（代码、名称、各维度收益、评分等）
3. **获取市场优选排行** — 加载推荐结果，格式化推荐榜单
4. **推送通知** — 依次执行（两条通道）：
   - **企业微信**：发送 Markdown 格式晚报（含基金数据表格 + 推荐排行）
   - **邮件**：若企业微信发送失败，则通过 SMTP 发送 HTML 格式邮件
5. **保存晚报 HTML** — 将渲染后的简报保存到 `.briefing_fund.html`，供管理页面展示

> 此步骤约耗时 **5~15 秒**（仅为文件读取和推送时间，不包含网络数据拉取）。

### Step 2 — 进度反馈
- 按钮变为「⏳ 晚报 {当前状态} {进度}%」
- 进度条同步更新，状态文字反映当前阶段：
  - `加载推荐数据...` → `整理 基金名(代码)` → `获取市场优选` → `推送中` → `完成`
- 整体耗时极短（5~15 秒），进度快速走完

### Step 3 — 完成后自动加载晚报内容
- 状态提示消失，按钮恢复「📊 生成晚报」
- 晚报内容区域自动刷新显示最新 HTML
- 任务概览区域同步刷新（更新运行状态）

## 页面加载行为

### 恢复上次晚报
- 页面加载时自动调用 `GET /api/briefing` 加载已保存的 `.briefing_fund.html`
- 如果文件存在则显示内容及日期（如 `2026-06-16`）
- 如果文件不存在则显示「☢ 晚报尚未生成」+ 提示文字

### 检测正在进行的任务
- 页面加载时检查心跳 `fund_briefing` 是否存活
- 如果存活，按钮自动置为禁用态，恢复进度显示
- 每 10 秒轮询一次，直到任务完成

### 自动刷新
- 全局心跳轮询（每 10 秒）检测 `.briefing_fund.html` 的修改时间
- 如果检测到文件变更（如定时任务 `fund_watch.timer` 在 15:30 生成），自动调用 `loadBriefing()` 刷新内容

## 非功能需求

| 需求 | 说明 |
|------|------|
| 幂等性 | 如果晚报任务已在运行，第二次点击提示「晚报生成任务正在运行中」，不重复启动 |
| 配置即时生效 | 当前设置的评分维度权重和筛选参数，必须在本轮晚报中生效 |
| 无状态刷新 | 刷新页面后，已生成的晚报内容恢复显示，正在进行的任务恢复进度跟踪 |
| 不重复推荐 | 晚报**不会**重新执行全市场推荐扫描，仅使用已有的推荐结果文件（需先运行「▶ 运行推荐」） |
| 推送兜底 | 企业微信推送失败时自动降级为邮件推送；两种通道都失败时仍保存 HTML |
| 推送流程透明 | 推送结果对用户不感知——按钮和进度条仅反映数据整理→推送的进度，推送是静默的 |
| 依赖推荐结果 | 晚报依赖于「▶ 运行推荐」的结果文件，若未运行过推荐则跳过生成 |

## 当前实现限制

| 限制 | 说明 | 影响 |
|------|------|------|
| 无实时数据 | 晚报使用推荐结果中的缓存数据，不包含当日涨跌幅和经理信息 | 日报中的「涨跌」列显示为 `-` |
| 无警报检测 | 跳过经理变更、规模翻倍、净值停滞等警报检查 | 晚报不包含警报信息 |
| 子进程崩溃无告警 | `stdout/stderr` 重定向到 `DEVNULL`，报错信息完全丢失 | 排查问题困难 |
| 推送失败静默 | 企业微信和邮件失败仅在日志中记录，用户无感知 | 用户可能以为推送成功 |
| 数据时效性 | 推荐结果可能不是当天运行，晚报显示的评分和收益数据可能与当天有偏差 | 建议在生成晚报前先运行推荐 |

## 不在此功能范围内

- ❌ 执行全市场基金推荐扫描（属于「▶ 运行推荐」按钮）
- ❌ 刷新「市场优选排行」和「自选基金完整数据」表格（同上）
- ❌ 盘中实时监控（属于定时任务 `fund_monitor.py`）
- ❌ 全球股市简报（属于定时任务 `global_briefing.py`）
- ❌ 配置推送目标（企业微信 Webhook 和邮件 SMTP 在 `.env` 中配置）
